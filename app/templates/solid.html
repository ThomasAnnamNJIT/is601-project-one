SOLID

SOLID is a programming methodology that was established by both Robert C. Martin and Michael Feathers. These SOLID principles allow developers principles to "...create more maintainable, understandable, and flexible software". The principles of SOLID that we will discuss in depth include the following:

  • Single Responsibility Principle
  • Open/Closed Principle
  • Liskov Substitution Principle
  • Interface Segregation Principle
  • Dependency Inversion Principle

SOLID Terminology

Single Responsibility
The Single Responsibility Principle states

...if we have 2 reasons to change for a class, we have to split the functionality in two classes. Each class will handle only one responsibility and if in the future we need to make one change we are going to make it in the class which handles it

Single Responsibility Principle https://www.oodesign.com/single-responsibility-principle.html
This is demonstrated within our Calculator example, where we have three subclasses which all extend from CalculatorOperation (i.e. abstract base class/interface), which are Addition, Subtraction, and Multiplication. All these subclasses have only single responsibility which is to either add, subtraction, or multiply, respectively. In essence, each class provides its own functionality without affecting the other classes.
Open/Closed
The Open/Closed Principle states

... the design and writing of the code should be done in a way that new functionality should be added with minimum changes in the existing code.

Open/Closed Principle https://www.oodesign.com/open-close-principle.html
In our Calculator example, we provide an abstract base class called CalculatorOperation, which is initialized with a set of values. From this, we provide a method called perform_operation, which is an abstract method, whose functionality is defined by its subclasses. From the earlier discussed in Single Responsibility, the subclasses involve Addition, Subtraction, and Multiplication. The Addition class will perform the functionality for adding numbers, the subtraction class will perform the functionality for subtracting numbers, and lastly, multiplication will perform the functionality for d multiplying numbers. With this design approach of a common base class, we can easily extend this to include other operations such as Division.
Liskov Substitution

... if a program module is using a Base class, then the reference to the Base class can be replaced with a Derived class without affecting the functionality of the program module.

Open/Closed Principle https://www.oodesign.com/open-close-principle.html
In our Calculator example, we see that perform_operation is an abstract method, which means that it must be implemented by all of it's subclasses. Therefore, if we had a single method that takes a CalculatorOperation has an argument, then any of its subclasses (i.e. Addition, Subtraction, or Multiplication) should all work as expected since it would call its implementation of the perform_operation method. This is also known as method overriding.
Interface Segregation

... that clients should not be forced to implement interfaces they don't use.

Open/Closed Principle https://www.oodesign.com/interface-segregation-principle.html
In our Calculator example, the only method that each subclass must implement is the perform_operation method. This ensures each subclass is performing that operation that is intended to perform. The subclass is not expected to implement/override any of the methods from its base class, if applicable.
Dependency Inversion

High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details. Details should depend on abstractions.

Dependency Inversion Principle https://www.oodesign.com/interface-segregation-principle.html
In our Calculator example, we have a constructor that accepts a list of tuples of numbers, and a single abstract method called perform_operation. All subclasses that extend Calculator must implement any of the abstract methods within the Calculator. In summation, the subclasses (low-level modules) depends on the details of the base class (high-level modules), however, the base class does not need to know the implementation of the low-level modules.
