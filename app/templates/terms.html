Class
A class is a blueprint that allow developers to build on abstractions they want to model for an object, to meet both business and functional requirements for their application.

This is to the equivalent to the arrange phase within AAA Testing. In other words, a fixture serves a purpose within your tests altogether. Type hint
Type hinting is defined as "...a formal solution to statically indicate the type of a value within your Python code" (https://realpython.com/lessons/type-hinting/). This is done typically with a methods arguments and it's return value. With type hinting, we can specify the type we're expecting (i.e int, str, List), default values for the parameters, and the return type. For example, in the above picture, we see that the expected return type associated with perform_operation is expected to be a float.
Type Casting
Type casting is defined as "...specify a type on to a variable". This is typically done with Python objects such as:
  • int()
  • str()
  • float()
For example to convert the number one into a float, we can simply do float(1), which would yield the result of 1.0.
Unit Test
A unit test is a python function that tests a particular piece of code (i.e a method), which can be thought of as the subject under test. Typically, with unit testing, developers mock out any external dependencies such as database operations or 3rd party API calls.

Principles of OOP

Encapsulation
Encapsulation means only methods in the class with the variables can refer to the instance variable. In other words, only certain fields of a given classes are able to called from other parts of your program. The three scopes that Python has includes public (this is the default), protected, and private.
Inheritance
Inheritance allows one class to inherit attributes and methods from one class to another.
Polymorphism
Polymorphism allows an object to take on many forms, this means for a given Python method, we can base in the base class as an argument, and when we use the base class and call a method, it will have a different behavior. This is shown in the Calculator example, with the perform_operations method between Addition, Subtraction, or Multiplication.
Abstraction
Classes are blueprints around the object(s) to be modeled. Abstraction is a way to provide common interfaces of methods and APIs, which are then implemented by their subclasses.

Overview of Design Patterns

A design pattern is defined as a "general repeatable solution to a commonly occurring problem in software design" (https://sourcemaking.com/design_patterns). The benefits of design pattern include the following:
  • Speed up of the development proces
  • Reduce bugs/defects earlier in the development process
  • Better code quality (i.e improves readability)
There are several types of design patterns which can be divided into the following categories:
  • Creational
  • Structural
  • Behavioral
Creational
Creational design patterns are designed for creating classes, some design patterns that are associated with creational design patterns include Singleton and Factory. The Singleton design pattern ensures only one instance of class can exist, this is most typically used within multithreading for synchronization purposes. The Factory design pattern ensures we can create any instance of derived class, this is most commonly done with a parent-child relationships, we will see how we use the factory design pattern in our Calculator class.
Structural
Structural design patterns are about composition, some design patterns that are associated with creational design patterns include Decorator and Proxy.
Behavioral
Behavioral design patterns are about class communication with one another, some design patterns that are associated with behavioral design patterns include Observer and Chain of Responsibility. The Observer design pattern notifies of changes to any number of classes, which is a highly effective when dealing with event driven architecture. The Chain of Responsibility deals with class delegation in terms of whether it wants to deal the request or not, or pass it to off to another handler. Overview of SOLID

SOLID is a programming methodology that was established by both Robert C. Martin and Michael Feathers. These SOLID principles allow developers principles to "...create more maintainable, understandable, and flexible software". The principles of SOLID include the following:

  • Single Responsibility Principle
  • Open/Closed Principle
  • Liskov Substitution Principle
  • Interface Segregation Principle
  • Dependency Inversion Principle
