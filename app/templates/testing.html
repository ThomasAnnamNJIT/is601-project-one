OOP Terminology

SOLID is a programming methodology that was established by both Robert C. Martin and Michael Feathers. These SOLID principles allow developers principles to "...create more maintainable, understandable, and flexible software". The principles of SOLID that we will discuss in depth include the following:

  • Single Responsibility Principle
  • Open/Closed Principle
  • Liskov Substitution Principle
  • Interface Segregation Principle
  • Dependency Inversion Principle

SOLID Terminology

Single Responsibility
The Single Responsibility Principle states

...if we have 2 reasons to change for a class, we have to split the functionality in two classes. Each class will handle only one responsibility and if in the future we need to make one change we are going to make it in the class which handles it

Single Responsibility Principle https://www.oodesign.com/single-responsibility-principle.html
This is demonstrated within our Calculator example, where we have three subclasses which all extend from CalculatorOperation (i.e. abstract base class/interface), which are Addition, Subtraction, and Multiplication. All these subclasses have only single responsibility which is to either add, subtraction, or multiply, respectively. In essence, each class provides its own functionality without affecting the other classes.
Open/Closed
The Open/Closed Principle states

... the design and writing of the code should be done in a way that new functionality should be added with minimum changes in the existing code.

Open/Closed Principle https://www.oodesign.com/open-close-principle.html
In our Calculator example, we provide an abstract base class called CalculatorOperation, which is initialized with a set of values. From this, we provide a method called perform_operation, which is an abstract method, whose functionality is defined by its subclasses. From the earlier discussed in Single Responsibility, the subclasses involve Addition, Subtraction, and Multiplication. The Addition class will perform the functionality for adding numbers, the subtraction class will perform the functionality for subtracting numbers, and lastly, multiplication will perform the functionality for d multiplying numbers. With this design approach of a common base class, we can easily extend this to include other operations such as Division.
Liskov Substitution

... if a program module is using a Base class, then the reference to the Base class can be replaced with a Derived class without affecting the functionality of the program module.

Open/Closed Principle https://www.oodesign.com/open-close-principle.html
In our Calculator example, we see that perform_operation is an abstract method, which means that it must be implemented by all of it's subclasses. Therefore, if we had a single method that takes a CalculatorOperation has an argument, then any of it's subclasses should be able to work when passed into the method.
Abstraction
As discussed above, classes are blueprints around the object(s) to be modeled. Abstraction is the same way, it provides a blueprint of common interfaces and api, which are then implemented by the subclasses. In other words, an abstract class cannot be instantiated on its own.

docker build -t [image_name] [path]
This allows you to build an image from the location of your Dockerfile and allows you to tag it with a name.
docker kill [container_id]
This allows you to destroy/remove a container without having to stop it beforehand
docker system prune
This command will destroy any containers that are stopped containers or remove any dangling images, which include those that are untagged.
docker run [image_id]
This command allows you to run a container after you have build an image from the docker build command; however, if you simply run this command without having built an image, it will build an image first, and then run the container for you automatically!
docker ps
This will list all of your running/stopped/exited containers
docker compose up
This commands allows you to bring multiple containers up at once within a service such as your application container and a db container, and then connecting your application to the DB. In addition, this will also create a network for you in order for your containers to communicate with each other. This is commonly defined within a docker.compose.yml file.
docker compose down
This commands will destroy all the containers associated with the service, along with any networks/volumes created

Project Setup

Bring Up Containers
To bring up the project please run docker compose up in the command line from the root of the project directory where your docker.compose.yml file is located, this will build/run the image associated with the Dockerfile and create a network configuration. Within the docker.compose.yml file, this application container maps from "80:5000", this mean you'll be able to access it on port 80, which would then map to port 5000 within the container.
Access the Container
To access the container, run docker exec -it [container_id] sh, this will essentially give you an interactive shell for the container
